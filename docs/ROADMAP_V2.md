# City1 v2 Roadmap

## Цель

Пересобрать проект в воспроизводимый и честный geospatial ML pipeline без разрыва между train и inference.

## Формулировка задачи

Так как у нас нет надёжного ground truth по населению на уровне сетки, v2 строится как:

- `proxy population surface estimation`
- `spatial disaggregation with city-level constraints`

Это означает:

- модель оценивает относительное распределение населения внутри города
- официальные totals по городу используются как ограничение, калибровка и weak supervision
- результаты нельзя позиционировать как перепись по квадратам

## Фаза 0. Freeze legacy

### Цель

Зафиксировать текущий проект как исследовательский архив.

### Действия

- не удалять старые ноутбуки и старые `.py`
- не строить новую логику поверх legacy-кода
- зафиксировать ключевые дефекты и ограничения

### Результат

- legacy остаётся источником контекста
- вся новая работа ведётся в `src/city1/`

## Фаза 1. Foundation

### Цель

Подготовить чистый каркас проекта.

### Действия

- ввести новую структуру директорий
- добавить `README`, `.gitignore`, базовые контракты данных
- формализовать набор обязательных признаков и артефактов

### Result

- понятный репозиторий
- минимальные правила и контракты

## Фаза 2. Data contracts

### Цель

Сделать данные проверяемыми, а не "как получилось".

### Действия

- описать схемы входных и выходных таблиц
- определить обязательные колонки и типы
- ввести проверки:
  - отсутствующие колонки
  - отрицательные площади/длины
  - пустые координаты
  - дубликаты зон

### Result

- любой dataset валидируется до обучения и до прогноза

## Фаза 3. Feature pipeline

### Цель

Построить единый модуль генерации признаков.

### Действия

- считать площади и длины только в проектной CRS
- отделить geometry CRS от display CRS
- реализовать настоящий `POI_Access_Index`
- унифицировать grid generation
- сохранить один общий pipeline для train и inference

### Result

- исчезает train/inference skew по признакам

## Фаза 4. Label strategy

### Цель

Честно определить, на чём обучается модель.

### Варианты

1. `Weak supervision`
   - официальная численность города
   - внутригородское распределение через эвристики/land-use/POI/buildings

2. `Semi-supervised`
   - если найдём частичные данные по районам или административным единицам

3. `Full supervised`
   - если появятся реальные spatial labels

### Result

- понятная научная постановка без ложных обещаний

## Фаза 5. Modeling

### Цель

Собрать воспроизводимое обучение.

### Действия

- baseline: Ridge / ElasticNet
- tree baseline: RandomForest
- boosting baseline: CatBoost
- spatial cross-validation
- holdout по городам
- логирование метрик и версий модели

### Result

- понятное сравнение моделей
- переносимость оценивается честно

## Фаза 6. Serving

### Цель

Сделать приложение тонким интерфейсом, а не местом, где живёт логика.

### Действия

- Streamlit вызывает функции из `src/city1/`
- добавить кэширование OSM-запросов
- ввести fallback на endpoints
- сделать понятные ошибки для пользователя

### Result

- UI остаётся стабильным
- логика живёт в тестируемом ядре

## Фаза 7. QA

### Цель

Сделать проект поддерживаемым.

### Действия

- unit tests
- regression tests
- smoke test на один город
- manual evaluation checklist

### Result

- изменения не ломают ядро незаметно

## Первые 5 правок с максимальным эффектом

1. Единый feature pipeline в проектной CRS.
2. Новый `POI_Access_Index`, считаемый по реальным POI.
3. Формальная label strategy с city-level constraints.
4. Полное выравнивание train и inference через общий код.
5. Жёсткие data contracts и валидация перед каждым шагом.
